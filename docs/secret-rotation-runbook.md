# Secret rotation + Sentry setup runbook

> Run this top-to-bottom. Each step rotates a credential that was
> committed to git in an earlier version of `.env.local`. Even though
> those commits aren't public today, the keys are exposed to anyone
> who ever clones or forks the repo — rotation kills them permanently.
>
> The final section sets up Sentry for production error tracking.

## Checklist

- [ ] Supabase service-role key rotated
- [ ] Resend API key rotated
- [ ] Google Maps API key rotated **and** locked to allowed referrers
- [ ] VAPID keys regenerated (already generated by Claude — paste them)

## 1. Supabase service-role key

**Why critical:** This key bypasses every Row Level Security policy.
A leaked one = full read/write across every table.

1. Open **supabase.com** → sign in → open Rajlo project.
2. Bottom-left ⚙ **Project Settings** → **API**.
3. Find `service_role` row → click reset / regenerate → confirm.
4. Copy the new key.
5. Update `.env.local`:
   ```
   SUPABASE_SERVICE_ROLE_KEY=<paste>
   ```
6. Vercel → Project Settings → Environment Variables → edit the same
   key → Save.
7. Vercel → Deployments → ⋯ → Redeploy latest.

The `anon` key in the same dashboard is meant to be public; don't
rotate it.

## 2. Resend API key

**Why critical:** Anyone with this can send email from your
@rajlo.com domain — phishing your own users.

1. Open **resend.com** → sign in → **API Keys**.
2. Find existing Rajlo key → ⋯ → **Revoke** (kills it instantly).
3. **Create API Key** → name `rajlo-prod` → permission `Sending access`.
4. Copy the new key — shown once only.
5. Update `.env.local`:
   ```
   RESEND_API_KEY=<paste>
   ```
6. Vercel → update + redeploy.

## 3. Google Maps API key (+ lockdown)

**Why critical:** Maps keys ride along in the browser bundle, so
they're discoverable. Without restrictions, a bot could rack up
five-figure bills overnight.

1. Open **console.cloud.google.com** → confirm Rajlo project in the
   top-left selector.
2. ☰ → **APIs & Services** → **Credentials**.
3. Click the existing API key → **REGENERATE KEY**.
4. **Don't save yet — lock it down first:**
   - **Application restrictions** → **HTTP referrers (websites)**.
   - Add:
     ```
     https://rajlo-v2.vercel.app/*
     https://*.vercel.app/*
     https://rajlo.com/*
     http://localhost:3000/*
     http://localhost:3001/*
     ```
   - **API restrictions** → **Restrict key** → tick:
     - Maps JavaScript API
     - Places API
     - Geocoding API
     - Routes API
   - Click **Save**.
5. Copy the regenerated key.
6. Update `.env.local`:
   ```
   NEXT_PUBLIC_GOOGLE_MAPS_API_KEY=<paste>
   ```
7. Vercel → update + redeploy.

## 4. VAPID keys

**Why critical:** These sign your web-push notifications. If leaked,
an attacker could forge fake push notifications to your users
("Your account was suspended — click here").

Fresh keys were generated in this chat. Use them. If you've lost
them, re-run:

```bash
npx web-push generate-vapid-keys --json
```

1. Update `.env.local`:
   ```
   VAPID_PUBLIC_KEY=<from chat>
   VAPID_PRIVATE_KEY=<from chat>
   NEXT_PUBLIC_VAPID_PUBLIC_KEY=<same as VAPID_PUBLIC_KEY>
   ```
2. Vercel → update all three variables + redeploy.

Caveat: rotating VAPID invalidates any existing push subscriptions.
Pre-launch this doesn't matter — no real users yet.

## Verification (post-rotation)

After the Vercel redeploy lands, smoke-test in production:

- [ ] Sign in → dashboards load → confirms Supabase keys work.
- [ ] Trigger a password-reset email → arrives → confirms Resend.
- [ ] Open a map (e.g. rider booking) → renders → confirms Maps.
- [ ] Enable push notifications → buzz arrives → confirms VAPID.

If any of these fail, the env var probably didn't persist or the
redeploy didn't pick it up — re-check the Vercel env var page and
manually redeploy.

## 5. Sentry setup

**Why:** without error tracking, you find out about production bugs
when angry users post on Twitter days later. Sentry catches every
crash in real time.

### Account + DSN

1. Open **sentry.io** → **Sign up** (free tier: 5k events/month,
   plenty for beta).
2. Email: `Raj@rajlo.com`.
3. Organization name: `Rajlo`.
4. When prompted to choose a platform → **Next.js**.
5. Project name: `rajlo-web`.
6. Skip the on-screen install wizard — Claude will do that part.
7. Project page → find **DSN** (looks like
   `https://xxxxxxx@oNNNN.ingest.sentry.io/NNNNNN`).
8. Copy it. Paste into the chat with Claude — that's the trigger
   for the SDK wiring.

### Recommended Sentry dashboard config

After the SDK is wired:

- Project → **Alerts** → create rule: "Send email when first event
  occurs in production". Catches the first real crash immediately.
- Settings → **Team Members** → invite owner / co-founders.
- Project → **Performance** → enable session replays (free tier
  includes some) — lets you see the exact UI state at the moment a
  user hit an error.
